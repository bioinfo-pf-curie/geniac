Develop a new release
=====================

The backbone of geniac consists of:

* A |nextflow|_ code (see :ref:`contributing-install-folder`),
* A |cmake|_ utilities (see in :ref:`contributing-install-folder`) which launches |nextflow|_ and install the analysis pipeline accoding to the list of options defined in :ref:`install-configure`.

In addition, a geniac Command Line Interface written in python (see :ref:`contributing-src-folder`) intends to simplify the use of geniac for users less familiar with |cmake|_ syntax. 

What does this mean for the developer? Well, you will have to dig into:

* The :ref:`contributing-install-folder` if you want to:

  - modify the content of files automatically generated by geniac (nextflow config files, container recipes)

  - add new config files
  
  - modify how containers are built and pushed on a registry

  - support new container technologies

* The :ref:`contributing-cmake-folder` if you want to:

  - change the installation process

  - add :ref:`install-custom-targets`

  - report new functionnalities implemented in the :ref:`contributing-install-folder`

* The :ref:`contributing-src-folder` if you want to:

  - improve the geniac Command Line Interface

  - report new functionnalities implemented in the :ref:`contributing-install-folder` and/or :ref:`contributing-cmake-folder`

Test a new release
==================

Test geniac CLI
---------------

Below, we assume that:

* ``${GENIAC_SRC_DIR}`` is the directory in which is located the geniac source code you are developping, typically this is the |geniacrepo|_

* ``${ANALYSIS_PIPELINE_DIR}`` is the directory in which is located the nextflow analysis pipeline you want to install, for example the |geniacdemo|_


First, install the geniac conda environment:

::

   # Create the geniac conda environment
   conda env create -f ${GENIAC_SRC_DIR}/environment.yml


Install your current geniac source code directory with ``pip``:

::

   # Activate the geniac conda environment
   conda activate geniac

   # Install/Update the geniac python package with pip
   pip install -U ${GENIAC_SRC_DIR}

   # The geniac python package is installed in
   GENIAC_INSTALL_DIR=$(pip show geniac | grep Location | awk '{print $2}')/geniac
   echo ${GENIAC_INSTALL_DIR}


Create a working directory with ``geniac init`` in the folder ``${WORK_DIR}``:

::

   # Initialization of a working directory
   # with the src and build folders
   geniac init -v -w ${WORK_DIR} ${ANALYSIS_PIPELINE_DIR}
   cd ${WORK_DIR}


Test the geniac CLI:

::

   # For example, test the linter
   geniac lint


.. note::

   Is it important to be aware that ``geniac init`` will create the folder ``${WORK_DIR}/src/geniac``. This folder is actually a copy of ``${GENIAC_INSTALL_DIR}/repo`` which contains the :ref:`contributing-install-folder`, the :ref:`contributing-cmake-folder`, the :ref:`contributing-assets-folder` and the main ``CMakeLists.cmake``. Note that if the ``geniac`` folder already exists in the ``${ANALYSIS_PIPELINE_DIR}``, it is replaced by ``${GENIAC_INSTALL_DIR}/repo``.


.. note::

   Whenever you modify the geniac source code in ``${GENIAC_SRC_DIR}``, and reinstall the geniac python package with ``pip install -U ${GENIAC_SRC_DIR}``, it will update the content of the folder ``${GENIAC_INSTALL_DIR}/repo``. However, your previous ``${WORK_DIR}/src/geniac`` will not be modified: its content will remain the same as the one when you launched the ``geniac init`` command the first time. To take into your modifications, you have to create another geniac working directory with ``geniac init``.
 

Non-regression tests
--------------------

Perform the following tests when developing a new geniac version:

::

    conda activate geniac
    pip uninstall -y geniac
    pip install -U ${GENIAC_SRC_DIR}
    geniac init -w ${WORK_DIR} ${ANALYSIS_PIPELINE_DIR}
    cd ${WORK_DIR}
    geniac lint
    geniac recipes -v
    geniac clean
    geniac configs -v
    geniac install . install -v
    geniac install . install -v --ap_install_singularity_images --test_stub_run
    geniac install . install -v --ap_install_docker_images --test_stub_run
    geniac install . install -v --ap_install_docker_images --test_stub_run --ap_docker_registry_push_repo='myregistry.org/' --ap_push_images
    geniac install . install -v --ap_install_podman_images --test_stub_run
    geniac install . install -v --ap_install_podman_images --test_stub_run --ap_docker_registry_push_repo='myregistry.org/' --ap_push_images



Checklist for a new release
---------------------------

* A new nextflow profile has been added:

  - For example, the profile ``apptainer.config`` is added. This new file ``apptainer.config`` must be listed in the ``GENIAC_CONFIG_FILES`` variable in the file ``src/geniac/cli/utils/base.py``. This variable provide the list of all the ``.config`` expected to be automatically generated by geniac. If these files have be committed in the source code of the nextflow pipeline to be installed, geniac will complain.
* A new option has been added in the cmake installation process:

  - For example, the option ``ap_linux_distro`` is added. :ref:`contributing-preload` to update the file ``data/cmake-init-default.cmake``.

  - Add the new option in the documentation (file ``docs/install.rst``)

* A new linux must be used as bootstrap for the container recipes:

  - For example, the linux distro ``almalinux:9.7`` must be used. Change the value ``ap_linux_distro`` in the file ``cmake/stepSetVariables.cmake``

* A new conda miniforge version must be used to build the conda environment inside for the container:
  - For example, the miniforge version ``25.11.0-1`` must be used. Change the value ``ap_conda_release`` in the file ``cmake/stepSetVariables.cmake``


Structure of the build directory tree
=====================================

Among the different files and folders created in the ``build`` directory, we just focus on the ``workDir`` folder that contains the files that are automatically generated by ``geniac``. For example:

::

   └── workDir
       └── results
           ├── conda
           │   └── environment.yml
           ├── conf
           │   ├── cluster.config
           │   ├── conda.config
           │   ├── docker.config
           │   ├── multiconda.config
           │   ├── multipath.config
           │   ├── path.config
           │   └── singularity.config
           └── singularity
               ├── deffiles
               │   ├── fastqc.def
               │   ├── helloWorld.def
               │   ├── multiqc.def
               │   ├── onlyLinux.def
               │   ├── python.def
               │   └── trickySoftware.def
               └── images
                   ├── alpine.sif
                   ├── fastqc.sif
                   ├── helloworld.sif
                   ├── multiqc.sif
                   ├── onlylinux.sif
                   ├── python.sif
                   └── trickysoftware.sif

The |nextflow|_ scripts that are used to generate the ``*.config`` files and containers are also located in the ``workDir`` folder. There are:

* ``docker.nf``: generates the |docker|_ containers and Dockerfiles.
* ``singularity.nf``: generates the ``*.config`` files, the |singularity|_ images and def files.
* ``nextflow.config``: is used by the above.

Generate preload cache with default values
==========================================

The installation of a pipeline can be made using ``cmake -C myfile.cmake`` which :ref:`install-configure-file`. The geniac repository provides the default file ``geniac/data/cmake-init-default.cmake`` with all the options which are available. This section describes how to generate the default file. 
In order to generate the pre-load script ``geniac/data/cmake-init-default.cmake`` to populate the *cmake* cache, use the ``geniac/cmake/initCmakePreload.sh`` as follows:

::

   # Assume that the variable SRC_DIR is set to the path of the source code of geniac
   export SRC_DIR="${HOME}/geniac"
   export WORK_DIR="${HOME}/tmp/"

   # Create the geniac conda environment
   conda env create -f environment.yml

   # Activate the geniac conda environment, otherwise cmake
   # would complain about missing dependencies
   conda activate geniac

   cd ${WORK_DIR}
   bash ${SRC_DIR}/cmake/initCmakePreload.sh ${SRC_DIR} > ${SRC_DIR}/data/cmake-init-default.cmake

Containers
==========

Build
-----

Inside the :ref:`contributing-install-folder` are located the  *nextflow* scripts allowing the automatic generation of recipes *def files* and *Dockerfiles* respectively. They also allow the building of the containers.

Options can be passed to these scripts and can be seen the ``geniac/install/nextflow.config.in``. 

These scripts are automatically called during the build step of the project (see :ref:`install-page`), thus you don't have to run them manually.

Labels
------

In order to tack from which repository and version the containers were built, some labels are added in the recipes. Here is an example from a singularity def file:

::

   %labels
       gitUrl https://github.curie.fr/bioinfo-pf-curie/geniac-demo.git
       gitCommit 80f6511b453a365be39.5bede6d79f0ce7253d16
   

Update the documentation
========================

Whenever required, update the documentation and build it locally to check the content:

::

    conda env create -f environment.yml
    conda activate readthedocs
    cd docs
    python -m sphinx -T -E -b html -d _build/doctrees -D language=en . _build/html
